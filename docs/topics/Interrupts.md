An **interrupt** is a signal sent to the CPU by hardware or software, indicating that an event requires immediate attention.  It causes the CPU to **pause** its current task, **save its state**, and execute a special function called an **Interrupt Service Routine (ISR)** to handle the event.  After the ISR completes, the CPU **resumes** the original task from where it was interrupted. 

There are two main types:

- **Hardware Interrupts**: Generated by external devices (e.g., keyboard press, timer, disk I/O completion). 
    
- **Software Interrupts**: Generated by programs to request operating system services (e.g., system calls) or by the CPU itself for exceptions (like division by zero). 
    

This mechanism is fundamental for multitasking, efficient I/O, and system responsiveness.

---

## Interrupt Descriptor Table (IDT) - Detailed Explanation

The Interrupt Descriptor Table (IDT) is a critical data structure on x86/x86_64 architectures that tells the CPU where to find interrupt and exception handlers. When an interrupt fires (hardware interrupt), exception occurs (page fault), or software interrupt executes (`int 0x80`), the CPU uses the IDT to locate and execute the appropriate handler function.

## What Problems Does the IDT Solve?

Without the IDT, the CPU wouldn't know what to do when:

- A keyboard key is pressed (hardware interrupt)
    
- You divide by zero (exception)
    
- A page fault occurs (memory protection violation)
    
- A syscall is invoked (software interrupt)
    

The IDT provides a mapping from interrupt vector numbers (0-255) to handler function addresses.

---
## IDT Format on x86_64

The IDT consists of **256 entries** (interrupt descriptors), each **16 bytes** (128 bits) long. Each entry is called a **gate descriptor** and comes in three types:

|Gate Type|Purpose|Use Case|
|---|---|---|
|**Interrupt Gate**|Hardware interrupts|Keyboard, timer, disk I/O|
|**Trap Gate**|Exceptions and software interrupts|Page faults, breakpoints, syscalls|
|**Task Gate**|Hardware task switching|Rarely used (not covered here)|

## Structure of a Single IDT Entry (x86_64)

```rust

#[repr(C, packed)]
#[derive(Debug, Clone, Copy)]
pub struct IdtEntry {
    // Low 16 bits of the handler address (offset)
    pointer_low: u16,
    
    // Code segment selector (points to GDT entry)
    segment_selector: u16,
    
    // IST (Interrupt Stack Table) offset (0-7, 0 means use default stack)
    ist: u8,
    
    // Flags + gate type (interrupt/trap) + descriptor privilege level (DPL)
    flags: u8,
    
    // Middle 16 bits of the handler address
    pointer_middle: u16,
    
    // High 32 bits of the handler address
    pointer_high: u32,
    
    // Reserved (must be 0)
    reserved: u32,
}
```


### Field-by-Field Breakdown

#### **1. Handler Address Fields (`pointer_low`, `pointer_middle`, `pointer_high`)**

These three fields combine to form a 64-bit function pointer:

text

Full address = (pointer_high << 32) | (pointer_middle << 16) | pointer_low

**Why split into 3 parts?** Historical compatibility with 32-bit x86 (which had 2 parts: low/high). x86_64 added the middle part for 64-bit addresses while maintaining ABI compatibility.

#### **2. Segment Selector (`segment_selector`)**

This is an index into the Global Descriptor Table (GDT) pointing to the code segment that the handler should run in. On x86_64, this is typically the kernel code segment (usually index 0x8 or 0x10).

```rust

// Example: GDT entry for kernel code at index 1 (0x8)
segment_selector: 0x08
```

#### **3. Interrupt Stack Table (IST) Field (`ist`)**

This field (3 bits) selects which stack to use when handling this interrupt. This is an x86_64-specific feature for handling kernel stack overflow:

- **0**: Use the current stack (default, no switch)
    
- **1-7**: Switch to a predefined IST stack from the Task State Segment (TSS)
    

**Use case**: Double fault handler needs its own stack because the kernel stack might be corrupted.

#### **4. Flags Field (`flags`)**

This 8-bit field controls critical behavior:

```text
Bit positions:
7 6 5 4 3 2 1 0
[P][DPL][0][G][1][1][0?]
```


|Bit(s)|Name|Meaning|
|---|---|---|
|7|Present (P)|Set to 1 if handler is valid (0 = unused entry)|
|6-5|DPL (Descriptor Privilege Level)|Minimum privilege level to call this gate (0 = kernel only, 3 = user code)|
|4|0|Must be 0 (storage segment indicator)|
|3|Gate Size (G)|x86_64: 1 = 64-bit gate, 0 = 32-bit gate (legacy)|
|2-0|Gate Type (software view)|110 = Interrupt Gate, 111 = Trap Gate|

**Gate Type Detail**:

- **Interrupt Gate (binary 1110)**: Clears IF flag (disables interrupts) automatically when entering handler
    
- **Trap Gate (binary 1111)**: Leaves IF flag unchanged (interrupts remain enabled)
    

## Complete IDT Entry Example

Here's how you would construct an IDT entry for a keyboard interrupt handler:

```rust
fn create_idt_entry(handler_addr: u64, ist_index: u8) -> IdtEntry {
    IdtEntry {
        pointer_low: (handler_addr & 0xFFFF) as u16,
        segment_selector: 0x08,  // Kernel code segment
        ist: ist_index & 0x7,    // Only use bits 0-2
        // P=1, DPL=0, G=1 (64-bit), Type=1110 (Interrupt Gate)
        flags: 0b1000_1110,      // = 0x8E
        pointer_middle: ((handler_addr >> 16) & 0xFFFF) as u16,
        pointer_high: ((handler_addr >> 32) & 0xFFFFFFFF) as u32,
        reserved: 0,
    }
}
```

## The IDT Register (IDTR)

The CPU has a special register called the **IDTR** that stores:

- **Base address**: Where the IDT starts in memory (64-bit)
    
- **Limit**: Size of IDT in bytes (16-bit) = (256 entries × 16 bytes) - 1 = 4095
    

The `lidt` instruction loads this register:

```rust
x86_64::instructions::tables::lidt(&IdtRegister {
    base: VirtAddr::new(idt_ptr as u64),
    limit: 256 * 16 - 1,
});

```

## Typical IDT Layout

Interrupt vectors are conventionally assigned:

|Vector Range|Type|Example|
|---|---|---|
|0-31|CPU Exceptions (non-maskable)|0: Divide by Zero, 14: Page Fault|
|32-47|Hardware IRQs (PIC)|32: Timer, 33: Keyboard|
|48-255|Software interrupts / Custom|128: Linux syscall (`int 0x80`)|

---

Each exception has a predefined IDT index. For example, the invalid opcode exception has table index 6 and the page fault exception has table index 14. Thus, the hardware can automatically load the corresponding IDT entry for each exception. The [Exception Table](https://wiki.osdev.org/Exceptions) in the OSDev wiki shows the IDT indexes of all exceptions in the “Vector nr.” column.
